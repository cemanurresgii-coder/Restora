## Architecture Overview

```
Browser
  └── React 19 (Vite 8)
        ├── Zustand stores  ←──── single source of truth for UI state
        │     └── call services
        └── Service layer   ←──── thin wrappers around Firebase SDK
              └── Firebase v12
                    ├── Auth  (email/password)
                    └── Firestore  (5 collections)
```

Components never import Firebase directly. All Firestore access goes through `src/services/`. All global state lives in `src/store/`. This separation makes the business logic easy to unit-test without mocking Firebase.

## Role-Based Access Control

Three roles are defined in `useAuthStore.js`:

| Role | Constant | Default route | What they can do |
|------|----------|---------------|-----------------|
| `customer` | `ROLES.CUSTOMER` | `/app` | Browse menu, place orders, make reservations, view restaurant info |
| `staff` | `ROLES.STAFF` | `/staff` | Manage reservations, update table status, advance orders through the kitchen queue |
| `manager` | `ROLES.MANAGER` | `/manager` | All staff actions + view KPI dashboard, analytics charts, advanced reports, and manage menu items |

**How roles are assigned:**
- New registrations always receive `customer` via `register()` in `useAuthStore`.
- To promote a user to `staff` or `manager`, manually update the `role` field in the `users` Firestore collection.

**How routes are protected:**
`<ProtectedRoute allowedRoles={['staff', 'manager']}>` checks `useAuthStore.role`. If the role is absent or not in `allowedRoles`, the user is redirected to the login page.

**Auth persistence:**
`useAuthStore` uses Zustand's `persist` middleware, storing `{ user, role }` in `localStorage` under the key `auth-storage`. The `initAuthListener()` function (called once in `App.jsx`) keeps this in sync with Firebase Auth's session state.

## Firestore Collections

### `users`

Created automatically on registration. Staff and manager roles must be set manually.

| Field | Type | Notes |
|-------|------|-------|
| `email` | `string` | Firebase Auth email |
| `role` | `string` | `"customer"` \| `"staff"` \| `"manager"` |

**Document ID:** Firebase Auth UID (`user.uid`)

---

### `menuItems`

Managed by the manager via the **Manage Menu** tab. Customers read from this collection when browsing the menu.

| Field | Type | Notes |
|-------|------|-------|
| `name` | `string` | Display name |
| `category` | `string` | Used for tab filtering (e.g. `"Starters"`, `"Mains"`, `"Desserts"`) |
| `price` | `number` | Price in the local currency unit |
| `description` | `string` | Short item description shown on the menu card |
| `emoji` | `string` | Single emoji used as the item's visual |
| `available` | `boolean` | `false` hides the item from customers |

**Document ID:** Auto-generated by Firestore (`addDoc`)

**Sort order:** Client-side `localeCompare` on `name` (no `orderBy` in query, so no composite index needed).

---

### `reservations`

Created by customers. Read and updated by staff and the manager.

| Field | Type | Notes |
|-------|------|-------|
| `uid` | `string` | Auth UID of the customer who booked (empty string for walk-ins) |
| `name` | `string` | Guest name |
| `email` | `string` | Contact email |
| `date` | `string` | ISO date `"YYYY-MM-DD"` |
| `time` | `string` | 24-hour time `"HH:MM"` |
| `guests` | `number` | Party size |
| `tableId` | `string` | References `tables` document ID (e.g. `"T3"`) |
| `zone` | `string` | Denormalised zone label (`"Indoor"`, `"Window"`, `"Outdoor"`) |
| `notes` | `string` | Special requests (empty string if none) |
| `status` | `string` | See status lifecycle below |
| `createdAt` | `Timestamp` | Firestore server timestamp |

**Document ID:** Auto-generated by Firestore

**Status lifecycle:**
```
pending ──► confirmed ──► completed
    └──────► cancelled
```

| Status | Meaning |
|--------|---------|
| `pending` | Awaiting staff confirmation |
| `confirmed` | Accepted by staff |
| `cancelled` | Rejected or cancelled by guest/staff |
| `completed` | Guest has left; table is free |

**Availability logic:** A table is considered booked for a slot if any reservation exists for that `date + time` with status **not** `cancelled`. The `getAvailableTables` derived selector in `useReservationStore` applies this filter client-side.

---

### `orders`

Created by customers after mock payment. Consumed by the kitchen queue in real time.

| Field | Type | Notes |
|-------|------|-------|
| `tableId` | `string` | Which table placed the order |
| `customerName` | `string` | Display name shown in the kitchen queue |
| `items` | `Array<OrderItem>` | See sub-schema below |
| `total` | `number` | Grand total (sum of `price × qty`) |
| `notes` | `string` | Order-level special requests |
| `status` | `string` | See status lifecycle below |
| `createdAt` | `Timestamp` | Used for FIFO sort in kitchen queue |
| `updatedAt` | `Timestamp` | Written on every `updateOrderStatus()` call |

**`OrderItem` sub-schema:**

| Field | Type | Notes |
|-------|------|-------|
| `id` | `string` | Source `menuItems` document ID |
| `name` | `string` | Denormalised item name |
| `price` | `number` | Price at time of order |
| `qty` | `number` | Quantity ordered |
| `customization` | `string` | Per-item special request (empty string if none) |

**Document ID:** Auto-generated by Firestore

**Status lifecycle:**
```
pending ──► preparing ──► ready ──► served
    └──────────────────────────────► cancelled
```

| Status | Meaning | Visible in kitchen queue? |
|--------|---------|--------------------------|
| `pending` | Order received, not yet started | Yes |
| `preparing` | Kitchen is working on it | Yes |
| `ready` | Ready to be picked up | Yes |
| `served` | Delivered to table | No |
| `cancelled` | Order voided | No |

**Real-time subscription:** `subscribeActiveOrders()` uses `where('status', 'in', ['pending', 'preparing', 'ready'])` to fetch only the active queue, sorted FIFO by `createdAt` client-side.

---

### `tables`

Seeded by `initializeTables()` (called when the staff dashboard mounts). Staff update statuses in real time via the Table Management panel.

| Field | Type | Notes |
|-------|------|-------|
| `tableNumber` | `number` | Display number; used for client-side sort |
| `capacity` | `number` | Maximum guests |
| `zone` | `string` | `"Indoor"` \| `"Window"` \| `"Outdoor"` |
| `status` | `string` | `"available"` \| `"reserved"` \| `"occupied"` |

**Document ID:** Fixed IDs `T1`–`T8` (set via `setDoc`, not `addDoc`)

**Idempotent seeding:** `initializeTables()` uses `setDoc(..., { merge: true })` so existing `status` values are never overwritten on re-seed. Safe to call on every mount.

**Canonical layout:**

| ID | # | Zone | Capacity |
|----|---|------|----------|
| T1 | 1 | Indoor | 2 |
| T2 | 2 | Indoor | 4 |
| T3 | 3 | Indoor | 6 |
| T4 | 4 | Window | 2 |
| T5 | 5 | Window | 4 |
| T6 | 6 | Outdoor | 4 |
| T7 | 7 | Outdoor | 8 |
| T8 | 8 | Outdoor | 10 |

## Service Layer

Services are pure async functions. They import `db` from `src/firebase/config.js` and use the Firebase v12 modular SDK. Components should never call Firebase directly — always go through a service.

```

Collection: "menuItems"
```

|----------|-----------|---------|-------|

| `addMenuItem` | `(item: object) => Promise<string>` | New document ID | |
| `updateMenuItem` | `(id: string, updates: object) => Promise<void>` | — | Partial update via `updateDoc` |
| `deleteMenuItem` | `(id: string) => Promise<void>` | — | Hard delete |

## Zustand Stores

All stores live in `src/store/`. Import with the default export, e.g.:

```js
import useOrderStore from '../store/useOrderStore'
const { activeOrders, subscribeActiveOrders } = useOrderStore()
```

### `useAuthStore`

**File:** `src/store/useAuthStore.js`  
**Persistence:** Yes — `user` and `role` are stored in `localStorage` (`"auth-storage"`).

#### State

| Key | Type | Initial | Description |
|-----|------|---------|-------------|
| `user` | `{ uid, email } \| null` | `null` | Currently signed-in user |
| `role` | `string \| null` | `null` | `"customer"` \| `"staff"` \| `"manager"` |
| `loading` | `boolean` | `false` | True while an auth operation is in flight |
| `error` | `string \| null` | `null` | Human-readable Turkish error message |

#### Actions

| Action | Signature | Notes |
|--------|-----------|-------|
| `login` | `(email, password) => Promise<void>` | Signs in, then fetches role from `users` collection |
| `register` | `(email, password) => Promise<void>` | Creates Auth user + `users` doc with `role: "customer"` |
| `logout` | `() => Promise<void>` | Signs out, clears store |
| `setUserFromAuth` | `(user, role) => void` | Called by `initAuthListener` — not for direct use |
| `clearError` | `() => void` | Resets `error` to `null` |

#### Module-level helper

```js
import { initAuthListener } from '../store/useAuthStore'
// Call once in App.jsx — returns the Firebase unsubscribe function
const unsub = initAuthListener()
```

### `useMenuStore`

**File:** `src/store/useMenuStore.js`  
**Persistence:** None — data re-fetched on mount.

#### State

| Key | Type | Initial | Description |
|-----|------|---------|-------------|
| `items` | `MenuItem[]` | `[]` | All menu items currently in memory |
| `loading` | `boolean` | `false` | |
| `error` | `string \| null` | `null` | |

#### Actions

| Action | Signature | Notes |
|--------|-----------|-------|
| `fetchItems` | `(category?: string) => Promise<void>` | Populates `items`; pass a category string to filter |
| `addItem` | `(item: object) => Promise<string>` | Writes to Firestore, then optimistically appends to `items` |
| `updateItem` | `(id, updates) => Promise<void>` | Writes to Firestore, then patches `items` in place |
| `deleteItem` | `(id) => Promise<void>` | Deletes from Firestore, then removes from `items` |
| `setItems` | `(items: MenuItem[]) => void` | Local-only override (no Firestore write) |

### `useReservationStore`

**File:** `src/store/useReservationStore.js`  
**Persistence:** None.

#### State

| Key | Type | Initial | Description |
|-----|------|---------|-------------|
| `reservations` | `Reservation[]` | `[]` | All reservations in memory |
| `loading` | `boolean` | `false` | |
| `error` | `string \| null` | `null` | |

#### Actions

| Action | Signature | Notes |
|--------|-----------|-------|
| `fetchReservations` | `() => Promise<void>` | Loads all reservations from Firestore |
| `createReservation` | `(data: object) => Promise<string>` | Writes to Firestore, then prepends new reservation to `reservations` |
| `updateStatus` | `(id, status) => Promise<void>` | Updates Firestore and patches local array |

#### Derived Selectors (call as functions, not actions)

```js
const { getAvailableTables, isTableOccupied } = useReservationStore()

// Returns Table[] — tables with capacity >= guests and no active booking at that slot
getAvailableTables(date, time, guests)

// Returns boolean — true if tableId has a pending or confirmed reservation
isTableOccupied(tableId)
```

These read from `reservations` in the store — call `fetchReservations()` first to ensure the data is fresh.

#### Exported constant

```js
import { TABLES } from '../store/useReservationStore'
// Array of 8 tables: [{ id, zone, capacity }, ...]
```

### `useOrderStore`

**File:** `src/store/useOrderStore.js`  
**Persistence:** None.

#### State

| Key | Type | Initial | Description |
|-----|------|---------|-------------|
| `orders` | `Order[]` | `[]` | Full order history (used by manager analytics) |
| `activeOrder` | `object \| null` | `null` | Customer's in-progress cart before submission |
| `activeOrders` | `Order[]` | `[]` | Kitchen's live queue (pending \| preparing \| ready) |
| `loading` | `boolean` | `false` | |
| `error` | `string \| null` | `null` | |

#### Actions

| Action | Signature | Notes |
|--------|-----------|-------|
| `setActiveOrder` | `(order) => void` | Stores a pre-submission cart object |
| `clearActiveOrder` | `() => void` | Resets `activeOrder` to `null` |
| `placeOrder` | `(orderData) => Promise<string>` | Calls `addOrder`, clears `activeOrder`, prepends to `orders` |
| `fetchOrders` | `() => Promise<void>` | Loads full order history; used by manager dashboard |
| `updateStatus` | `(id, status) => Promise<void>` | Updates Firestore and patches `orders` in place |
| `getOrdersByStatus` | `(status) => Order[]` | Derived filter over `orders` (not `activeOrders`) |
| `subscribeActiveOrders` | `() => unsubscribe` | Opens real-time listener; populates `activeOrders`; returns cleanup fn |

**Kitchen queue usage pattern:**

```js
// Inside a component's useEffect
const { subscribeActiveOrders } = useOrderStore()

useEffect(() => {
  const unsub = subscribeActiveOrders()
  return unsub // cleanup on unmount
}, [])
```

## End-to-End Workflows

### Table Booking Workflow

**Actor:** Customer  
**Components:** `TableReservation.jsx`  
**Stores/Services:** `useReservationStore`, `reservationService.js`

```
1. Customer opens /app → TableReservation tab
2. Customer fills date, time, guest count → form submits
3. Component calls useReservationStore.getAvailableTables(date, time, guests)
   └── Reads store's reservations array
   └── Filters out occupied tables (active booking at same date+time)
   └── Filters out tables with capacity < guests
4. Available tables are displayed; customer selects one
5. Component calls useReservationStore.createReservation(data)
   └── Calls reservationService.createReservation()
   └── Writes to Firestore with status: "pending"
   └── Prepends new reservation to store (optimistic)
6. Staff dashboard receives the new pending reservation
   └── subscribePendingReservations() fires callback
7. Staff confirms or cancels → updateReservationStatus()
```

---

### In-App Ordering & Payment Workflow

**Actor:** Customer  
**Components:** `MenuContent.jsx`, `MockPaymentModal.jsx`  
**Stores/Services:** `useOrderStore`, `orderService.js`

```
1. Customer browses /app → Menu tab
2. Customer adds items to cart (local state in MenuContent)
3. Customer opens CartDrawer → clicks "Pay & Place Order"
4. MockPaymentModal opens (z-[60], above drawer's z-50)
5. Customer enters card details (client-side only, no real charge)
6. After 1.5s simulated processing → "Payment Approved!" shown
7. After additional 0.6s delay → onSuccess() fires
8. MenuContent.handlePaymentSuccess() calls handlePlaceOrder()
   └── Calls useOrderStore.placeOrder(orderData)
       └── orderData = { tableId, customerName, items, total, notes }
       └── items = cart.map(i => ({ id, name, price, qty, customization }))
   └── addOrder() writes to Firestore with status: "pending"
9. Kitchen queue (OrderQueue.jsx) receives the order in real time
   └── subscribeActiveOrders() fires
   └── New-order toast fires (skipped on initial load via null sentinel)
10. Cart is cleared; toast.success('Order placed!') shown to customer
```

**Cart total formula:** `cart.reduce((sum, i) => sum + i.price * i.qty, 0)`  
**Item count formula:** `cart.reduce((sum, i) => sum + i.qty, 0)`

---

### Staff Kitchen Queue Workflow

**Actor:** Staff  
**Components:** `OrderQueue.jsx` (Kanban board), `StaffDashboard.jsx`  
**Stores/Services:** `useOrderStore`, `orderService.js`

```
1. Staff opens /staff → Kitchen Queue tab
2. OrderQueue.jsx mounts → calls useOrderStore.subscribeActiveOrders()
   └── Firestore listener fires immediately with pending|preparing|ready orders
   └── Orders sorted FIFO by createdAt
3. Orders are displayed in 3 Kanban columns:
   Pending → Preparing → Ready
4. Staff clicks advance button on an order card
   └── Calls useOrderStore.updateStatus(id, nextStatus)
   └── updateOrderStatus() writes status + updatedAt to Firestore
   └── Firestore listener fires → store re-renders column
5. When staff marks order as "Served" from the Ready column:
   └── status becomes "served"
   └── subscribeActiveOrders filter excludes it → order disappears from board
6. StaffDashboard keeps a count badge on the Kitchen Queue tab
   └── Reads useOrderStore.activeOrders.length
   └── Capped display at "9+"
```

**Order status advancement:**
- `pending` → `preparing` (staff starts cooking)
- `preparing` → `ready` (food is done)
- `ready` → `served` (food delivered to table)

---

### Manager Analytics Workflow

**Actor:** Manager  
**Components:** `ManagerDashboard.jsx`, `DashboardKPIs.jsx`, `AnalyticsCharts.jsx`  
**Services:** `getOrders()`, `getReservations()`, `getTables()`

```
1. Manager opens /manager → Dashboard tab
2. ManagerDashboard.jsx mounts → fires Promise.all([getOrders(), getReservations(), getTables()])
3. Three pure aggregation functions compute chart data:
   computeRevenueTrend(orders)
   └── Groups served orders by day over last 14 days
   └── Returns [{ date, revenue }] sorted oldest-first
   computePopularItems(orders)
   └── Counts qty sold per item across all served orders
   └── Returns top 8, sorted by total descending
   computePeakHours(reservations)
   └── Counts active (non-cancelled) reservations per time slot
   └── Returns top 8 slots, reversed for recharts horizontal bar rendering
4. DashboardKPIs renders 4 cards:
   - Total Revenue (sum of served order totals)
   - Today's Orders (orders where createdAt date === today)
   - Active Reservations Today (pending+confirmed, date === today)
   - Table Utilization % (occupied+reserved / total tables)
5. Manager can click "Refresh" to re-fetch all data
```

## Real-Time Subscription Pattern

All real-time features follow the same convention — use `useEffect` with cleanup:

```js
import { useEffect } from 'react'
import useOrderStore from '../store/useOrderStore'

function MyComponent() {
  const subscribeActiveOrders = useOrderStore((s) => s.subscribeActiveOrders)

  useEffect(() => {
    const unsubscribe = subscribeActiveOrders() // opens Firestore listener
    return unsubscribe                           // closes it on unmount
  }, [])
}
```

**Available real-time subscriptions:**

| Function | Source file | What it streams |
|----------|-------------|-----------------|
| `subscribeActiveOrders(cb)` | `orderService.js` | Orders with status `pending\|preparing\|ready` |
| `subscribeTableUpdates(cb)` | `tableService.js` | All 8 table documents |
| `subscribeUserReservations(uid, cb)` | `reservationService.js` | Reservations belonging to a user |
| `subscribePendingReservations(cb)` | `reservationService.js` | All pending reservations |
| `useOrderStore.subscribeActiveOrders()` | `useOrderStore.js` | Wraps service; populates `activeOrders` in store |

**Why no `orderBy` in real-time queries?**  
Combining `where('status', 'in', [...])` with `orderBy('createdAt')` requires a Firestore composite index. To avoid the index setup step for new developers, all real-time queries use only the `where` filter and sort client-side after the snapshot arrives.

## Design Conventions

**Tailwind CSS v4 design tokens** (defined in `src/index.css`):

| Token | Value | Usage |
|-------|-------|-------|
| `--color-brand-500` | `#f97316` | Primary orange — buttons, active tabs, icons |
| `--color-brand-50/100/600` | orange scale | Backgrounds, hover states, dark text |

`bg-brand-500`, `text-brand-600`, `border-brand-200` etc. are available as utility classes throughout the app.

**recharts color note:** Tailwind classes don't apply inside SVG elements. All recharts `stroke` and `fill` values use the hardcoded hex `#f97316` (brand-500) directly.

**z-index layering:**

| Layer | z-index | Element |
|-------|---------|---------|
| Sticky navbar | `z-40` | Top navigation bar |
| Tab bar | `z-40` | Manager/Staff tab row |
| Filter bar | `z-30` | Menu category filter |
| Cart drawer | `z-50` | Customer cart slide-out |
| Payment modal | `z-[60]` | MockPaymentModal overlay |

**Toast notifications** (react-hot-toast):
- Global `<Toaster position="top-right">` in `App.jsx`
- Success toasts: brand-orange icon, 4 000 ms duration
- New-order toasts in `OrderQueue.jsx`: custom amber styling `{ background: '#fff7ed', color: '#c2410c', border: '1px solid #fed7aa' }`
- False-positive prevention: `prevCountRef = useRef(null)` — when `null`, the initial snapshot is loading and no toast is fired.

**Unit tests** (`src/__tests__/`):
- Business logic is extracted as pure functions in each test file — no Zustand, Firebase, or React imports.
- Run: `npm test` (single run) or `npm run test:watch` (watch mode).
- Framework: Vitest + jsdom + @testing-library/jest-dom.
